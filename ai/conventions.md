# Project Conventions

## 공통 규칙

- 기존 폴더 구조를 우선한다.
- 기존 네이밍 컨벤션을 따른다.
- 기존 주석은 보존한다.
- 관련 없는 파일은 수정하지 않는다.
- 불필요한 대규모 리팩토링은 하지 않는다.
- 포맷팅만 변경된 diff를 만들지 않는다.
- 패키지 매니저는 기존 프로젝트 기준을 따른다.

## Backend 규칙

- Controller는 요청/응답 처리에 집중한다.
- 비즈니스 로직은 Service에 둔다.
- DB 접근은 Repository 또는 ORM 계층에서 처리한다.
- DTO validation을 유지한다.
- Exception 형식은 기존 프로젝트와 맞춘다.
- Module 구조는 기존 프로젝트 구조를 따른다.
- 외부 API 연동 로직은 별도 Client/Provider/Service로 분리한다.

## DB 규칙

- DB 변경 시 migration 필요 여부를 먼저 판단한다.
- 운영 데이터 손실 가능성이 있으면 반드시 경고한다.
- soft delete가 있는 테이블은 조회 조건에 누락하지 않는다.
- 조회 조건이 많은 필드는 인덱스 필요 여부를 검토한다.
- 트랜잭션이 필요한 로직인지 확인한다.

## API 규칙

- 기존 응답 구조를 임의로 변경하지 않는다.
- 필수값 검증을 명확히 한다.
- 인증/권한이 필요한 API는 Guard 적용 여부를 확인한다.
- Swagger 문서가 있는 프로젝트라면 API 변경 시 문서 반영을 고려한다.

## Git 규칙

- 하나의 작업은 가능한 작은 단위로 나눈다.
- 관련 없는 파일 수정 금지.
- 작업 후 수정 파일 목록과 이유를 정리한다.
